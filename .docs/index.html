Cursor Skills

Cursor Skills is a repository of reusable AI skill playbooks, agent-team definitions, and a small FastAPI demo app. It also includes a self-contained evaluation harness that stages repeatable scenarios in eval/sandbox/ so each skill can be validated with both automated checks and human quality review.

Architecture

The repository is organized around three complementary workflows. First, the skills/ directory defines process-heavy SKILL.md documents that teach an AI how to execute tasks like walkthroughs, visual reviews, and wiki generation. Second, .claude/agents/ provides role-specific agent prompts (builder, reviewer, documenter) that coordinate larger multi-agent pipelines. Third, eval/ acts as a deterministic testing harness where each skill gets a controlled sandbox state before output is validated.

The included app/ service is a minimal FastAPI task API used as a concrete code target for tests and examples. In contrast, eval/fixture-app/ is a separate Flask application used exclusively for scenario-driven skill evaluation. This split is intentional: the demo app shows a clean implementation baseline, while the fixture app can be reset and mutated per scenario without affecting the root project.

Note: Two API stacks coexist by design: app/ (FastAPI demo) and eval/fixture-app/ (Flask scenario fixture). Most skill validation runs against the sandbox copy of the fixture app, not the root FastAPI app.

Key Flows

Skill Evaluation Lifecycle

The core lifecycle is designed to be reproducible. A scenario setup script creates a clean git state in eval/sandbox/, the user runs the target skill in Cursor against that sandbox, and then a validator enforces structural expectations. A companion checklist captures quality criteria that scripts cannot reliably evaluate.

run-eval setup → open sandbox → run skill → validate.sh + checklist

Demo API Request Flow

The FastAPI app itself is deliberately small: app/main.py wires configuration and routers, app/routes/tasks.py maps HTTP concerns to helper functions, and app/models/task.py owns in-memory state and CRUD primitives. This keeps route handlers thin and makes behavior easy to test with TestClient.

API Routes

The demo service mounts routes under /api. The table below documents the behavior currently implemented in app/routes/tasks.py and how each endpoint maps to model helpers.

Agent Teams

Under .claude/agents/, the repository defines specialized teammate roles and a workflow catalog in WORKFLOWS.md. The intent is to move from ad-hoc prompting to repeatable collaboration where each role has a clear output contract: builder writes code and progress.md, reviewer writes .review/review.md, and documenter writes .docs/index.html.

AgentPrimary ResponsibilityKey Artifact
builderChunk plan, implement incrementally, track statusprogress.md
reviewerAudit git diff and explain risks with diagrams.review/review.md
documenterSynthesize architecture into single-page docs.docs/index.html

The hook script at .claude/hooks/on-task-complete.sh provides guardrails by checking for tests on implementation tasks, minimum review quality on review tasks, and wiki size for documentation tasks. This gives lightweight quality gates without requiring a full CI pipeline.

Evaluation Framework

The eval/ system is where most operational rigor lives. Each scenario has a setup script that constructs the exact repo state expected by a skill, a validator that checks structural output, and a checklist for manual review. This combination catches both hard failures (missing files/sections) and softer quality issues (accuracy, clarity, consistency).

review-diff

Injects uncommitted Flask app changes and expects .review/review.md with diagrams and diff narrative.

walkthrough

Creates a feature branch with auth changes and evaluates conversational file-by-file explanation quality.

create-wiki

Starts from a clean codebase and checks .docs/index.html structure, content breadth, and styling signals.

incremental-implementation

Seeds a plan file and verifies chunk tracking, code deltas, query/filter behavior, and expanded tests.

Testing

Tests in tests/test_tasks.py target the FastAPI demo using fastapi.testclient.TestClient. The fixture resets in-memory state before and after each test via reset_store(), which keeps test order from leaking state between cases.

Coverage currently emphasizes core behavior: create, list, read existing/missing tasks, and delete existing/missing tasks. For production-facing evolution, the next step would be adding tests for concurrent access assumptions, persistence migration behavior, and richer validation rules.

Configuration

Configuration is intentionally lightweight. app/core/config.py defines a Settings class with app_name and app_version using pydantic-settings, and app/main.py injects those values into FastAPI metadata. 

from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    app_name: str = "Task Tracker API"
    app_version: str = "0.1.0"

There is no deployment manifest or CI file in the repository yet. That keeps setup friction low for skill development, but it also means environment parity and release checks are currently manual.
