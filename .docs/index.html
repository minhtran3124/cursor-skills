Task API

A minimal, self-contained FastAPI application that provides CRUD operations for a task list. There is no database — all state lives in an in-memory list, which makes the server fast to start and trivial to run during development and demos.

FastAPI Pydantic v2 pydantic-settings v2 Uvicorn pytest Python 3.10+

Overview

Task API is a demo FastAPI project generated by an automated agent team. Its purpose is to illustrate a clean four-layer project layout — scaffold, config/models, routing, entry point — while keeping the implementation small enough to read in one sitting.

The application exposes four HTTP endpoints that let clients create tasks, list all tasks, fetch a single task by its numeric ID, and delete a task. Tasks are plain objects with an auto-incrementing integer ID, a non-empty title string, and a boolean done flag. State is held in a module-level list inside app/routes/tasks.py and is lost when the process restarts — intentionally so, since persistence was out of scope.

The project reached its current state after three rounds of code review. The two issues surfaced in review — a silent pydantic-settings v2 compatibility bug in the config module and a missing min_length constraint on the task title — were both resolved before approval. A full test suite of seven pytest cases accompanies the production code and all tests pass.

Quick Start

The project has no build step and no database to provision. Three commands get you from a fresh checkout to a running server. 

# 1. Install dependencies (Python 3.10+ required)
pip install -r requirements.txt

# 2. Start the development server with auto-reload
uvicorn app.main:app --reload

# 3. Open interactive API docs in your browser
# http://127.0.0.1:8000/docs

Once the server is running, FastAPI automatically serves an OpenAPI UI at /docs (Swagger UI) and an alternative ReDoc UI at /redoc. Both are generated from the Pydantic models and endpoint decorators — no extra configuration is required.

Running the test suite

Tests use FastAPI's built-in TestClient, which wraps httpx and does not require a running server. Run all seven tests from the project root: 

pytest tests/ -v

Overriding configuration via environment variables

The app reads its name and version from environment variables prefixed with APP_. You can override the defaults before starting the server: 

APP_NAME="My Task App" APP_VERSION="1.0.0" uvicorn app.main:app --reload
Note: The APP_ prefix is applied via pydantic-settings v2's SettingsConfigDict. The settings are read once at import time and cached in the module-level settings singleton in app/core/config.py.

Architecture

The application follows a straightforward layered design. An HTTP client talks to app/main.py, which is the only FastAPI application instance. At startup, the entry point reads the Settings singleton from app/core/config.py and registers the tasks router from app/routes/tasks.py. All request handling, data validation, and in-memory storage are confined to the router module and the Pydantic models it imports.

There is deliberately no service layer or repository abstraction. For a project of this scale, a service layer would add indirection without benefit. If the project were to grow — adding a real database, multiple resource types, or business rules — the natural next step would be to extract a app/services/task_service.py and move the storage and mutation logic out of the router.

Request lifecycle

HTTP Request → FastAPI routing → Pydantic validation → Route handler → JSON response

FastAPI handles JSON deserialization and Pydantic model instantiation before the route handler is ever called. If the request body fails validation (for example, an empty title), FastAPI short-circuits with a 422 Unprocessable Entity response and the handler is never invoked. This means the handler can assume its inputs are already valid.

In-memory storage model

Storage is a module-level list _tasks: list[Task] and a counter _next_id: int, both declared at the top of app/routes/tasks.py. The counter starts at 1 and is incremented after each successful POST. The global _next_id declaration inside create_task is required because Python's scoping rules treat an assignment to a bare name as a local binding unless the function explicitly opts in to the enclosing module scope.

Warning: All task data is lost when the server restarts. IDs also do not recycle — if you create task 5 and then delete it, the next task created will have ID 6, not 5. This is fine for a demo but would need to be addressed before any production use.

Project Structure

The project uses a four-package layout that separates configuration, domain models, HTTP routing, and the application entry point into distinct directories. Each directory contains an __init__.py that registers it as a Python package, enabling absolute imports like from app.models.task import Task.

Tech Stack

The dependency surface is intentionally minimal. Every package in requirements.txt is a direct dependency — there are no optional extras pulled in beyond what is needed for the server and configuration.

Package Version pin Role
fastapi >=0.110.0 Web framework. Handles routing, dependency injection, OpenAPI generation, and request/response validation via Pydantic.
uvicorn[standard] >=0.29.0 ASGI server. The [standard] extra includes httptools and watchfiles for better performance and --reload support.
pydantic-settings >=2.2.0 Settings management. Reads app configuration from environment variables using pydantic-settings v2's SettingsConfigDict API.
Note: pydantic itself is a transitive dependency (pulled in by both fastapi and pydantic-settings) but is not explicitly pinned in requirements.txt. If Pydantic ever releases a v3 with breaking changes, an unconstrained resolver could pull it in. A future hardening step would be to add pydantic>=2.0,<3 explicitly.

API Routes

All routes are registered under the /tasks prefix by the APIRouter(prefix="/tasks", tags=["tasks"]) declaration in app/routes/tasks.py. The router is mounted without an additional prefix in app/main.py, so the final paths are exactly as shown below.

FastAPI generates interactive documentation for all four endpoints automatically. Visit http://127.0.0.1:8000/docs when the server is running to explore and test them in your browser.

GET
/tasks
Returns the full list of tasks. Response is an array of Task objects. Returns an empty array when no tasks exist — never a 404.
POST
/tasks
Creates a new task. Expects a JSON body matching TaskCreate: a non-empty title string and an optional done boolean (defaults to false). Returns the created Task with its assigned id. Status code 201 Created. An empty or missing title returns 422 Unprocessable Entity.
GET
/tasks/{task_id}
Returns a single Task by its integer ID. If no task with that ID exists, returns 404 Not Found with body {"detail": "Task not found"}. The lookup is an O(n) linear scan of the in-memory list.
DELETE
/tasks/{task_id}
Deletes the task with the given ID. Returns 204 No Content on success with an empty body (as per HTTP spec). Returns 404 Not Found if the ID does not exist. The in-memory list is mutated via list.pop(index) after a linear scan.

Example interactions

# Create a task
curl -X POST http://localhost:8000/tasks \
  -H "Content-Type: application/json" \
  -d '{"title": "Write documentation"}'
# → 201 {"id": 1, "title": "Write documentation", "done": false}

# List all tasks
curl http://localhost:8000/tasks
# → 200 [{"id": 1, "title": "Write documentation", "done": false}]

# Fetch one task
curl http://localhost:8000/tasks/1
# → 200 {"id": 1, "title": "Write documentation", "done": false}

# Delete a task
curl -X DELETE http://localhost:8000/tasks/1
# → 204 (empty body)

# Attempt to create a task with an empty title
curl -X POST http://localhost:8000/tasks \
  -H "Content-Type: application/json" \
  -d '{"title": ""}'
# → 422 {"detail": [...]}

Data Models

The project defines two Pydantic models in app/models/task.py. Keeping input and output representations separate is a deliberate choice: it gives the API freedom to evolve the stored shape (for example, adding server-generated timestamps) without leaking those internals to callers.

TaskCreate — request body model

Used as the body type for POST /tasks. FastAPI deserializes the incoming JSON and instantiates this model before the route handler is called. If validation fails, the request never reaches application code. 

class TaskCreate(BaseModel):
    title: str = Field(..., min_length=1)  # required; empty string is rejected
    done: bool = False                     # optional; defaults to False

The Field(..., min_length=1) constraint was added after the first code review round. Without it, a POST body of {"title": ""} would have been accepted and stored — a silent data quality bug.

Task — full representation model

Returned by all endpoints that produce task data. Extends the fields of TaskCreate with a server-assigned id. This model is also used as the element type for the in-memory list _tasks: list[Task] in the router module. 

class Task(BaseModel):
    id: int       # auto-assigned by the router; starts at 1, never reused
    title: str
    done: bool = False

Configuration

Application settings are managed by app/core/config.py using pydantic-settings. The file exposes a module-level settings singleton that is imported by app/main.py to name and version the FastAPI instance. 

from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_prefix="APP_")

    app_name: str = "Task API"
    app_version: str = "0.1.0"

settings = Settings()

The model_config = SettingsConfigDict(env_prefix="APP_") line is the pydantic-settings v2 idiom. The v1 idiom — an inner class Config with env_prefix = "APP_" — is silently ignored by v2, which means env vars would never be applied. This was one of the bugs caught and fixed during code review.

SettingEnvironment variableDefaultDescription
app_name APP_NAME Task API Shown in the OpenAPI title field and the Swagger UI header.
app_version APP_VERSION 0.1.0 Shown in the OpenAPI version field.
Note: pydantic-settings also supports loading values from a .env file. To enable that, add env_file=".env" to the SettingsConfigDict call. No .env file is present in this project by default.

Testing

The test suite lives in tests/test_tasks.py and uses FastAPI's TestClient, which internally uses httpx to make synchronous requests against the ASGI app without a real network socket. This means tests are fast and self-contained — no server process needs to be started.

State isolation fixture

Because the task store is module-level state, tests that run sequentially would otherwise bleed state into each other (a task created in test 1 would still be visible in test 2). The reset_tasks autouse fixture solves this by directly mutating the router module's globals before each test: 

@pytest.fixture(autouse=True)
def reset_tasks():
    tasks_module._tasks.clear()
    tasks_module._next_id = 1
    yield

The autouse=True flag means every test in the file gets this fixture automatically — no test function needs to request it by name. The yield makes this a teardown fixture in principle, but in practice the reset happens before each test (on the pre-yield side) rather than after, which is the correct approach: it ensures a clean state regardless of whether the previous test passed or failed.

Known Tradeoffs

Several design decisions were made deliberately to keep the project small. They are worth documenting so that anyone extending the project understands what would need to change.

Storage in the router module

_tasks and _next_id live in app/routes/tasks.py alongside the route handlers. This couples storage to HTTP concerns in a way that makes isolated unit testing of the storage logic harder — you have to import the router module and reach into its globals to reset state, which is what the test fixture does. A cleaner alternative would be a dedicated app/core/store.py module that owns the list and counter, or using FastAPI's Depends mechanism to inject the store as a dependency.

No persistence

All data is in process memory. A server restart wipes every task. Adding SQLite via SQLAlchemy or a hosted Postgres via asyncpg would be the natural next step for a production system. The Pydantic models and route signatures would not need to change — only the storage implementation inside the router would be replaced.

No UPDATE endpoint

There is no PUT /tasks/{id} or PATCH /tasks/{id} endpoint. The done field cannot be toggled once a task is created. Adding a PATCH endpoint would require a new input model (e.g., TaskUpdate with all optional fields) and a handler that merges the patch into the existing task.

O(n) lookups

GET /tasks/{id} and DELETE /tasks/{id} both scan the entire list to find a task by ID. For a small demo list this is negligible. A dictionary keyed by ID would make lookups O(1) at the cost of a slightly more complex data structure.

pydantic not pinned directly

pydantic is a transitive dependency of both fastapi and pydantic-settings but is not listed in requirements.txt. If a future resolver pulls in a hypothetical Pydantic v3 with breaking changes, the app could break without any change to requirements.txt. Adding pydantic>=2.0,<3 would guard against this.
